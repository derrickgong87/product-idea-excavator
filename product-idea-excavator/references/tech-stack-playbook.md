# 技术栈 Playbook

技术建议要服务产品阶段和团队能力。不要为了先进而复杂。

## 技术判断原则

- 成熟默认：除非新技术带来明确产品优势，否则优先选择团队能掌控、社区成熟、容易招聘和维护的技术。
- 可逆优先：早期尽量选择可替换方案。不可逆或迁移成本高的决策要明确写入 PRD。
- 完整而不过度：如果补齐错误处理、日志、权限、测试和后台只多一点成本，就不要把它们推到以后。
- 先验证价值，再扩架构：不要为还没被验证的需求提前搭复杂平台。
- 用户结果优先：技术栈服务用户结果，不服务技术炫技。
- 可观测是产品能力：关键路径必须能看到成功、失败、延迟、成本和用户行为。

## 决策变量

先问清：

- 目标平台
- 团队熟悉技术
- 上线时间
- 预算
- 用户规模
- 数据敏感度
- 是否需要合规
- 是否需要支付
- 是否需要实时协作
- 是否需要第三方集成
- 是否需要后台运营
- 是否需要 AI
- 哪些选择可逆，哪些不可逆
- 哪些能力可以先人工或半自动验证

## Web SaaS MVP

默认建议：

- 前端：Next.js + TypeScript + React + Tailwind
- 后端：Next.js API routes / server actions，复杂后端再拆服务
- 数据库：Postgres
- ORM：Prisma 或 Drizzle
- 鉴权：Clerk 或 Auth.js
- 支付：Stripe
- 分析：PostHog
- 部署：Vercel
- 监控：Sentry + 平台日志

适合：快速验证、B2B SaaS、AI Web 产品、创始团队。

质量闸门：

- 关键用户路径有埋点。
- 错误状态对用户可见。
- 管理员能处理异常数据。
- 重要数据有导出能力。
- 核心功能有自动化测试或至少明确测试计划。

## AI Web 产品

默认建议：

- Web 层：Next.js + TypeScript
- 模型层：Vercel AI SDK 或类似抽象层
- 数据库：Postgres
- 向量检索：pgvector 起步，规模变大再考虑专用向量库
- 文件存储：S3/R2/Blob
- 任务队列：需要长任务时引入队列
- 评测：建立小型 eval set
- 成本：记录每次调用 token、耗时、模型、结果质量

关键取舍：先做质量闭环，再追求复杂 Agent。

早期不要急着上复杂编排。先证明：

- 用户愿意提供真实输入。
- 输出能节省真实时间。
- 用户愿意编辑、采纳或分享结果。
- 错误可被发现和修正。

## AI Agent 产品

必须设计：

- 工具调用权限
- 任务状态机
- 人工确认节点
- 失败重试
- 审计日志
- 回滚或撤销
- 成本上限
- 可观测性
- eval harness

不要让 Agent 在没有边界的情况下直接执行高风险动作。

Agent 的默认安全策略：

- 建议优先于执行。
- 草稿优先于发布。
- 人工确认优先于全自动。
- 可撤销优先于不可逆。
- 记录每一步优先于黑盒执行。

## RAG 产品

必须设计：

- 数据源
- 文档切分
- 元数据
- 权限过滤
- 召回策略
- 重排
- 引用来源
- 答案置信度
- 评测集
- 用户反馈闭环

MVP 可用 Postgres + pgvector 起步。

## 内部工具

默认建议：

- 极早期：Retool / Appsmith / 飞书多维表格
- 定制化：Next.js + Postgres + RBAC
- 必备：操作日志、权限、导入导出、异常处理

原则：不要把内部工具做成漂亮但难维护的 SaaS。

内部工具的常见失败不是界面不够漂亮，而是权限、异常处理、操作日志和数据修正能力缺失。

## 移动产品

默认建议：

- MVP：React Native / Expo
- 强原生能力：Swift / Kotlin
- 后端：Postgres + API
- 推送：平台推送服务
- 分析：移动端事件埋点

判断是否必须原生：

- 是否重度使用系统能力
- 是否需要极致性能
- 是否要求复杂离线体验

## 企业 B2B

优先考虑：

- SSO/SAML
- RBAC
- 审计日志
- 数据隔离
- Webhook
- 队列
- 监控
- 备份
- 合规路径

不要用只适合消费级 MVP 的方案承诺企业级能力。

如果第一批客户还不是企业，不要提前承诺完整企业能力。可以在 PRD 中分层：

- MVP：基础团队权限。
- V1：审计日志、角色权限。
- 企业版：SSO/SAML、数据隔离、合规流程。

## Marketplace

技术重点：

- 身份认证
- 支付/托管
- 评价系统
- 争议处理
- 供需匹配
- 运营后台
- 风控

冷启动比技术更关键，PRD 中必须写运营策略。
