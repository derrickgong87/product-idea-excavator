# PRD 模板与质量标准

使用本模板生成最终 PRD。不要机械填空，要根据访谈信息裁剪、补充和标注假设。

## PRD 标题

格式：`# [产品名称] PRD`

## 1. 一句话总结

用一句话说明：

- 给谁用
- 解决什么问题
- 用什么核心方式解决
- 带来什么结果

质量标准：不要写成口号，要能让工程师和设计师理解产品边界。

## 2. 背景与机会

说明：

- 用户为什么现在需要这个产品
- 现有方案为什么不够好
- 机会窗口是什么

标注哪些是用户已确认，哪些是助手推断。

## 3. 目标用户与画像

包含：

- 第一批用户
- 使用者、付费者、决策者
- 用户当前工作流
- 用户痛点频率与严重程度

如果用户画像还不清楚，明确写入“待确认”。

## 4. 问题定义

用具体场景描述问题：

- 触发场景
- 用户当前行为
- 痛点成本
- 为什么值得解决

避免只写“效率低”“体验差”。

## 5. 当前替代方案

列出用户现在可能使用的：

- 人工流程
- Excel / Notion / 文档
- 微信 / 飞书 / Slack
- 竞品
- 内部系统
- 干脆不解决

分析替代方案的不足。

## 6. 需求证据

区分信号强弱：

- 已确认强证据
- 中等证据
- 弱信号
- 尚未验证的假设

重点写清：

- 用户是否已经付费、预付、投入真实数据、替换现有流程或主动寻找方案
- 用户如果没有这个产品，会不会着急或产生明确损失
- 还缺什么证据

## 7. 最窄切口

说明：

- 第一批用户
- 最小可验证结果
- 最小可交付形态
- 为什么这个切口比完整平台更适合第一版
- 哪些范围故意不做

## 8. 核心价值主张

说明产品承诺：

- 更快
- 更便宜
- 更专业
- 更可靠
- 更自动化
- 以前做不到

写出用户第一次觉得“值了”的时刻。

## 9. 产品目标与非目标

目标：

- MVP 要验证什么
- 用户要完成什么
- 业务要证明什么

非目标：

- 明确第一版不做什么
- 哪些诱人功能要延后

## 10. MVP 范围

用 P0 / P1 / P2 划分：

- P0：没有就无法验证价值
- P1：增强体验，但可延后
- P2：后续路线图

每个功能要写：

- 用户价值
- 简要描述
- 优先级理由

## 11. 核心用户路径

写出至少一个完整路径：

1. 入口
2. 初始状态
3. 用户输入或选择
4. 系统处理
5. 结果呈现
6. 用户确认、编辑、分享或导出
7. 后续留存动作

要包含失败路径和恢复方式。

## 12. 功能需求

每个功能建议包含：

- 功能名
- 用户故事
- 触发条件
- 主要流程
- 边界条件
- 验收标准

## 13. 非功能需求

根据产品需要覆盖：

- 性能
- 可用性
- 安全
- 隐私
- 合规
- 可扩展性
- 可观测性
- 多端适配

## 14. UX 要求

覆盖：

- 首次使用体验
- 空状态
- 加载状态
- 错误状态
- 权限拒绝
- AI 不确定或失败时的表达
- 信任与解释

## 15. 数据需求

说明：

- 核心对象
- 数据来源
- 数据字段
- 数据权限
- 数据生命周期
- 导入导出

## 16. AI / 模型需求

如果适用，说明：

- AI 承担的角色
- 输入
- 输出
- 模型能力要求
- 是否需要 RAG / Agent / 工具调用 / 多模态
- 评测指标
- 幻觉和错误处理
- 人工审核机制
- 成本和延迟约束

## 17. 技术栈建议

格式：

```text
推荐技术栈：
- 前端：
- 后端：
- 数据库：
- 鉴权：
- AI / 模型层：
- 文件 / 对象存储：
- 支付：
- 通知：
- 分析：
- 部署：
- 监控与日志：
- 后台运营：

为什么适合：
- ...

主要取舍：
- ...

MVP 可以先怎么做：
- ...

什么时候应该换方案：
- ...
```

## 18. 系统架构建议

写清楚：

- 主要模块
- 数据流
- 第三方集成
- 异步任务
- 权限边界
- 日志与审计

## 19. 权限与角色

如适用，定义：

- 普通用户
- 管理员
- 团队成员
- 审核者
- 外部协作者

## 20. 集成需求

列出：

- 必需集成
- 可选集成
- Webhook / API
- 导入导出

## 21. 运营后台需求

说明运营人员需要：

- 查看什么
- 修改什么
- 审核什么
- 处理什么异常

## 22. 指标体系

至少包含：

- North Star Metric
- 激活指标
- 留存指标
- 转化指标
- 质量指标
- 业务指标
- 风险指标

AI 产品还要考虑：

- 准确率
- 人工修正率
- 幻觉率
- 平均延迟
- 单次任务成本

## 23. 边界情况与失败状态

列出：

- 无数据
- 数据错误
- 权限不足
- 第三方集成失败
- AI 失败
- 用户中途退出
- 支付失败
- 重复提交

## 24. 前提挑战与替代方案

写出：

- 当前方案依赖的 3-5 个关键前提
- 哪个前提风险最大
- 至少 2 个可选路径
- 推荐路径和理由
- 用户明确选择或仍待确认的地方

## 25. 风险与待验证假设

至少区分：

- 产品风险
- 技术风险
- 市场风险
- 数据风险
- 合规风险

每个风险写可能影响和验证方式。

## 26. Roadmap

建议分：

- MVP
- V1
- V2
- 长期方向

## 27. 产品经理建议

写出你作为顶级 PM 的判断：

- 最推荐的切入点
- 最应该砍掉的范围
- 最关键的验证实验
- 最值得提前设计的能力
- 最容易误判的指标或假设

## 28. 待确认问题

明确列出仍需要用户确认的问题，不要隐藏不确定性。
