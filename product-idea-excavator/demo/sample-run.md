# 样例试跑：AI 客户反馈挖掘器

## 测试输入

用户：

> 我想做一个 AI 工具，帮产品团队自动整理客户反馈，最后生成 PRD。你来问我问题。

## Skill 应答片段

明白，这个 idea 的核心不是“整理文本”，而是把分散的客户反馈转成产品决策材料，最后沉淀成 PRD。  

我先暂时把它理解为一个面向产品团队的 AI 工作流工具：输入来自客户沟通、销售记录、客服工单或访谈纪要，系统做聚类、提炼痛点、识别机会，再辅助生成需求文档。

我的产品经理建议是：第一版不要直接做“全自动生成 PRD”，而是先做“反馈收集 -> 聚类 -> 洞察卡片 -> PRD 草稿”的半自动流程。原因是 PRD 需要产品判断，如果一开始全自动，用户会不信任；半自动更容易验证价值，也更容易让用户参与修正。

但我先不把“产品团队觉得有用”当成需求证据。这个方向要成立，必须证明团队现在已经为整理反馈付出了真实成本。

第一个关键问题：这些客户反馈最初主要来自哪里？比如飞书/微信群、Slack、客服系统、CRM、销售笔记、邮件、用户访谈录音，还是已经整理好的表格？

## 假设用户回答 1

用户：

> 主要来自飞书群、销售的聊天记录，还有一些用户访谈录音。目标用户应该是 10-50 人的小型 B2B SaaS 团队。

## Skill 应答片段

这已经很清楚了：第一批用户不是所有产品团队，而是小型 B2B SaaS 团队，反馈来源分散在飞书、销售聊天和访谈录音里。这里的核心痛点大概率是“信息散、没人系统整理、产品决策靠记忆和感觉”。

我建议 MVP 先不要做所有渠道集成。第一版可以先支持两类输入：手动粘贴聊天记录 + 上传访谈转写文本。飞书自动同步可以放到 V1，因为它会增加权限、集成和数据安全复杂度。

这里我想做一个前提挑战：如果这些团队只是“偶尔觉得反馈散”，但没有固定整理动作、没有会议决策成本、没有错过需求的后果，那产品可能只是 nice-to-have。

下一个问题：你见过的最强需求证据是什么？比如某个团队每周花几小时整理、因为漏看反馈做错优先级、愿意付费、还是已经在用表格/文档手工维护？

## 假设用户回答 2

用户：

> 有个团队每周产品例会前会让 PM 手工翻销售聊天和访谈记录，大概要花半天。他们抱怨很多重复反馈没人系统整理，但还没付费，只说如果能生成可用洞察愿意试。

## Skill 应答片段

这个证据比“大家感兴趣”强，因为已经有固定工作流和明确时间成本：每周半天。它还不是最强证据，因为没有付费或预付，但足够支持一个窄 MVP 验证。

我会把当前前提写成三条：

1. 第一批用户是 10-50 人 B2B SaaS 团队里的产品负责人或 PM。
2. 他们的现状成本是每周手工翻销售聊天和访谈记录，为产品例会准备输入。
3. MVP 的核心价值不是自动写完整 PRD，而是在例会前更快产出可信的机会洞察。

如果这三条成立，最窄切口应该是“例会前反馈洞察包”，而不是完整客户反馈平台。

下一个问题：用户使用这个工具后，最希望得到的第一个高价值结果是什么？是反馈分类、需求优先级、用户痛点总结、机会洞察，还是可以直接进入评审的 PRD 草稿？

## 假设用户回答 3

用户：

> 最想要机会洞察和 PRD 草稿。PRD 不需要完全自动，但要有一个很好的初稿。

## Skill 应答片段

这说明 AI 的角色应该是“产品分析助理”，不是完全替代 PM。它要把原始反馈变成可讨论的机会洞察，再生成 PRD 初稿，让 PM 修改确认。

这里我会建议把核心输出拆成两层：

- 洞察层：痛点聚类、代表性原话、影响用户数、严重程度、建议优先级。
- 文档层：基于选中的洞察生成 PRD 草稿，明确标注“来自用户反馈的证据”和“AI 推断”。

下一个关键问题是技术落地：第一版你希望它是 Web SaaS，还是先做成飞书/Slack 机器人？这会影响输入方式、权限、留存和技术栈。

## 假设用户回答 4

用户：

> 先做 Web SaaS，后面再接飞书。技术上希望能快速上线验证。

## Skill 应答片段

很好。Web SaaS 优先是合理的：它能更快验证核心流程，也能避开早期复杂的企业 IM 权限集成。飞书集成可以作为 V1 的留存增强。

技术上我会倾向：

- 前端：Next.js + TypeScript + React + Tailwind
- 后端：Next.js API / server actions 起步
- 数据库：Postgres
- 文件存储：对象存储，用于访谈文本和附件
- AI 层：先用通用大模型 + 结构化输出；后续再加 RAG 和评测集
- 向量检索：MVP 可先不做复杂向量库，先做项目内文本聚类与引用；需要跨历史反馈检索时再加 pgvector
- 分析：PostHog
- 部署：Vercel

我的建议是先验证“上传/粘贴反馈 -> 生成洞察 -> 选择洞察 -> 生成 PRD 草稿”这条闭环。  

我会保留两个替代路径：

- 最小验证版：只做粘贴/上传文本、洞察卡片、PRD 草稿导出。最快验证质量。
- 稳健 MVP 版：加项目空间、历史洞察、团队评论。更像正式产品，但会慢一点。
- 长期版本：接飞书、CRM、客服系统，形成持续反馈库。适合验证留存后再做。

我的建议是先走最小验证版，因为现在最大风险是洞察质量，而不是集成能力。

下一步我想确认 MVP 成功标准：用户做到什么动作，我们才认为这个产品真的有价值？是上传反馈、生成洞察、导出 PRD、邀请团队评审，还是一周内复用第二次？

## 假设用户确认挖掘完成

用户：

> 先这样，没有其他补充了，可以总结 PRD。

## Skill 应自动输出的 PRD 摘要示例

# AI 客户反馈挖掘器 PRD

## 1. 一句话总结

面向 10-50 人 B2B SaaS 产品团队的 AI 产品分析工具，帮助团队把分散在聊天记录和访谈文本中的客户反馈转化为机会洞察，并生成可编辑的 PRD 草稿。

## 2. MVP 范围

P0：

- 手动粘贴聊天记录
- 上传访谈转写文本
- AI 反馈聚类
- 洞察卡片生成
- 基于选中洞察生成 PRD 草稿
- 标注原始证据和 AI 推断
- 导出 Markdown / 文档

P1：

- 飞书集成
- 团队协作评论
- 历史洞察库
- 优先级评分规则自定义

非目标：

- 第一版不做完整 CRM
- 第一版不做全自动产品决策
- 第一版不做多渠道实时同步
- 第一版不承诺 PRD 无需人工修改

## 3. 推荐技术栈

- 前端：Next.js + TypeScript + React + Tailwind
- 后端：Next.js API routes / server actions
- 数据库：Postgres
- ORM：Prisma 或 Drizzle
- 文件存储：S3/R2/Blob
- AI 层：通用大模型 + 结构化输出
- 检索：MVP 暂不引入复杂向量库；V1 可加 pgvector
- 鉴权：Clerk 或 Auth.js
- 分析：PostHog
- 部署：Vercel
- 监控：Sentry

## 4. 关键风险

- AI 洞察质量不足，PM 不信任。
- 原始反馈来源分散，导入成本影响激活。
- PRD 草稿如果太泛，会被认为只是模板生成。
- 飞书/销售聊天记录涉及敏感数据，后续需要权限和合规设计。

## 5. 产品经理建议

第一版最应该验证的不是“能不能生成漂亮 PRD”，而是“AI 提炼出的洞察是否让 PM 感到比自己手动整理更快、更准、更有证据”。建议围绕洞察质量建立评测集和用户反馈闭环。
